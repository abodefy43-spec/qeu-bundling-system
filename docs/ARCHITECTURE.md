# ARCHITECTURE.md

## High-level architecture
QEU Bundling is a staged offline pipeline plus a presentation layer.

```text
Raw Orders + Reference Data
    -> Phase 00-05 feature/metadata build
    -> Phase 06 candidate generation/ranking
    -> Phase 07 model training
    -> Phase 08 candidate scoring/output shaping
    -> Phase 09 validation/optimization artifacts
    -> output/* + data/processed/* artifacts
    -> Flask dashboard (people-based recommendations)
```

## Subsystem breakdown
- **CLI Layer** (`src/qeu_bundling/cli`)
  - Command dispatch for runs, serving, usage, review, quality evaluation.
- **Config Layer** (`src/qeu_bundling/config`)
  - Path resolution and canonical folder layout.
- **Pipeline Layer** (`src/qeu_bundling/pipeline`)
  - Phases 00-09 for ingestion, feature building, candidate selection, modeling, scoring, optimization.
- **Core Utilities** (`src/qeu_bundling/core`)
  - Run manifests, quality gates, review exports, feedback memory, usage summaries.
- **Presentation Layer** (`src/qeu_bundling/presentation`)
  - Flask app, person recommendation orchestration, semantic engine, dashboard i18n.
- **Review Layer** (`src/qeu_bundling/review`)
  - Structured curated feedback loader and lookup builder.

## Folder-by-folder explanation
- `src/qeu_bundling/cli`: entrypoint (`__main__.py`) and subcommand definitions.
- `src/qeu_bundling/config`: filesystem contracts (`paths.py`).
- `src/qeu_bundling/pipeline`: numbered phases with deterministic artifact outputs.
- `src/qeu_bundling/core`: cross-phase support (manifests, quality checks, review pack).
- `src/qeu_bundling/presentation`: web app and recommendation-time serving logic.
- `src/qeu_bundling/review`: feedback class normalization (`strong/staple/weak/trash`).
- `templates` + `static`: dashboard UI contract.
- `tests`: unittest-based regression suite.
- `data/raw`, `data/processed`, `data/reference`: source/intermediate/reference data.
- `output`: models, scored outputs, manifests, logs, review artifacts.

## Data flow
### Pipeline data flow
```text
order_items.csv
    -> phase_00_extract_campaigns (campaign pairs, product pictures)
    -> phase_01_load_data (filtered_orders.pkl + recipe artifacts)
    -> phase_02_embeddings (embeddings + top similarity)
    -> phase_03_copurchase (copurchase_scores.csv)
    -> phase_04_categories (product_categories.csv)
    -> phase_05_recipe_scoring (product_recipe_scores.csv)
    -> phase_06_bundle_selection (person_candidate_pairs.csv + diagnostics)
    -> phase_07_train_models (model *.pkl + metrics)
    -> phase_08_predict (person_candidates_scored.csv)
    -> phase_09_optimize (quality checks + artifact summary)
```

### Presentation-time data flow
```text
latest manifest + person_candidates_scored.csv + filtered_orders.pkl
    -> load_bundle_view()
    -> build/load person profiles
    -> build_recommendations_for_profiles()
    -> optional dashboard translation layer
    -> dashboard.html render
```

### Final selection hygiene (person-level)
- Final selection is centralized in `presentation/person_predictions.py` via `_select_final_candidates_for_person()`.
- This final stage applies, in order:
  - consumption compatibility hard blocks (`_consumption_pair_reject_reason`)
  - strict no-item-reuse within one person (an item can appear once across selected bundles)
  - cross-person pair repetition penalty (session/run scoped)
  - tuna soft penalty (final-stage ranking only; tuna is not banned)
  - fallback motif caps and nonfood cap
- Lane labels (`meal/snack/occasion/nonfood`) remain visible metadata; final selection does not enforce lane quotas.

### Pricing contract
- Bundle paid-item pricing is fixed to **80% discount on profit margin**:
  - `final_paid_price = purchase_price + 0.2 * (sale_price - purchase_price)`
- Shared constant/helpers live in `core/pricing.py`:
  - `FIXED_MARGIN_DISCOUNT_PCT`
  - `price_paid_and_free_items_fixed_margin()`
- API (`api/server.py`), phase-08 output shaping (`pipeline/phase_08_predict.py`), and dashboard rendering paths consume the same fixed-margin rule.

## Request lifecycle
1. Client requests `/`.
2. `presentation/app.py` loads run status and session-scoped person state.
3. `bundle_view.load_bundle_view()` resolves latest scored candidate artifact.
4. Missing recommendations are generated by `build_recommendations_for_profiles()`.
5. Translation pass enriches display-only Arabic fields.
6. Template renders cards and feedback forms.
7. Feedback POST (`/people/feedback`) appends row to `data/processed/person_feedback.csv`.

## State management approach
- **Persistent state**: filesystem artifacts and CSV/JSON outputs.
- **Ephemeral state**: in-memory `_PERSON_STATES` dict keyed by session token, TTL-pruned.
- **Run metadata state**: manifests in `output/runs/<run_id>/run_manifest.json` and `output/latest/latest_manifest.json`.

## Database/storage model
No database layer detected.
Storage is file-based:
- tabular: CSV/PKL,
- vectors/matrices: NPY/NPZ,
- metadata: JSON,
- models: PKL.

## External integrations
- `sentence-transformers` for product name embeddings.
- `xgboost` and `scikit-learn` for training/inference.
- `deep-translator` for product/dashboard translation with graceful fallback.

## Auth model
No user authentication/authorization model is implemented in Flask routes.
Session usage is for UI state continuity only.

## Background jobs/workers
- Dashboard refresh uses a background thread (`run_service.py`) that shells out to `python -m qeu_bundling.cli run full`.
- No dedicated task queue (Celery/RQ/etc.) found.

## Build/runtime model
- Runtime entrypoint: `python -m qeu_bundling.cli ...`.
- Package shim (`qeu_bundling/__init__.py`) allows running without editable install.
- CI: GitHub Actions workflow runs dependency install and `unittest` suite.

## Testing strategy
- Primary framework: `unittest`.
- Unit tests cover:
  - semantic engine rules,
  - person prediction gates/ranking/fallback,
  - run manifest/seed behavior,
  - phase-specific helpers,
  - review pack output and UI behavior.
- Canonical command:

```bash
python -m unittest discover -s tests -p "test_*.py"
```

## Architectural risks and weak points
1. **Artifact coupling risk**
   - Many modules assume specific CSV columns and file locations.
2. **Single-file complexity risk**
   - `person_predictions.py` is large and dense; high regression risk when editing.
3. **In-memory UI state risk**
   - Session recommendation state is process-local and reset on restart.
4. **Quality gate run-fail risk**
   - Runners raise when critical quality gates fail, interrupting quick/full workflows.
5. **Translation dependency risk**
   - External translator availability impacts display translation speed/quality.
6. **Encoding debt**
   - Some historical files/artifacts still show mojibake text artifacts.
